# MVP Test Plan

Дата: 2026-05-22

Этот документ фиксирует ручные сценарии для первого закрытого тестирования.
План нужен для двух групп:

- внутренний smoke test перед выдачей доступа;
- ранние пользователи alpha.

## 1. Цель тестирования

Подтвердить, что MVP:

- запускается;
- позволяет пользователю загрузить документы и получить ответы;
- корректно изолирует данные пользователей;
- не дает обычному пользователю server-level доступ;
- корректно считает usage и лимиты.

## 2. Тестовые роли

- `admin` - пользователь, чей email входит в `MULTI_USER_ADMIN_EMAILS`;
- `user_a` - обычный пользователь;
- `user_b` - второй обычный пользователь для проверки изоляции.

## 3. Тестовые данные

Минимальный набор:

- 2-3 небольших `.md` или `.txt` файла для `user_a`;
- 1-2 отдельных файла для `user_b`;
- 1 документ с повторяющимися или спорными фактами для проверки conflicts.

Рекомендации:

- использовать не секретные документы;
- не начинать alpha с больших PDF;
- не мешать документы разных тем в одном первом тесте.

## 4. Pre-flight checklist

Перед ручным тестированием:

- приложение запущено;
- `GET /api/health` отвечает;
- `GET /api/health` возвращает `mode=multi_user`;
- `.env` заполнен;
- admin email задан;
- тестовая база чистая или состояние понятно;
- известные ограничения доступны тестировщику.

## 5. Smoke: Admin

### A1. Регистрация admin

Шаги:

1. Открыть UI в чистом browser context или инкогнито.
2. Убедиться, что виден экран login/register.
3. Зарегистрироваться на email из `MULTI_USER_ADMIN_EMAILS`.
4. Открыть профиль или выполнить `GET /api/auth/me`.

Ожидание:

- браузер не показывает основной wiki UI до входа;
- регистрация успешна;
- выдан bearer token;
- `is_admin=true`.

### A2. Доступ к admin API

Шаги:

1. Выполнить `GET /api/admin/settings`.
2. Выполнить `GET /api/admin/settings/language`.
3. Выполнить `GET /api/admin/settings/test`.

Ожидание:

- все запросы доступны админу;
- ответы не содержат ошибок доступа.

## 6. Smoke: Обычный пользователь

### U1. Регистрация и вход

Шаги:

1. Открыть UI в чистом browser context.
2. Убедиться, что виден экран login/register.
3. Зарегистрировать `user_a`.
4. Выйти.
5. Войти снова.

Ожидание:

- до входа основной UI недоступен;
- register/login/logout работают;
- после logout старый token больше невалиден.

### U2. Ограничение admin доступа

Шаги:

1. Под `user_a` вызвать `GET /api/admin/settings`.
2. Под `user_a` вызвать `GET /api/admin/settings/test`.

Ожидание:

- ответ `403`.

### U3. Загрузка документов

Шаги:

1. Под `user_a` загрузить 2-3 документа.
2. Проверить список raw файлов.

Ожидание:

- файлы загружены;
- raw список содержит только документы текущего пользователя.

### U4. Ingest

Шаги:

1. Запустить ingest для загруженного документа.
2. Дождаться результата.

Ожидание:

- ingest завершается успешно;
- создаются или обновляются wiki страницы;
- при конфликте он попадает в conflict flow, а ingest не падает целиком.

### U5. Wiki

Шаги:

1. Открыть дерево wiki.
2. Открыть страницу.
3. Выполнить поиск.

Ожидание:

- дерево видно;
- страницы читаются;
- поиск возвращает только данные текущего workspace.

### U6. Chat

Шаги:

1. Задать 3 вопроса по загруженным документам.
2. Открыть history sessions.
3. Открыть конкретную session.

Ожидание:

- ответы приходят;
- есть citations;
- сессии видны только текущему пользователю;
- история не смешивается с другими пользователями.

### U7. Usage

Шаги:

1. До первого chat/ingest сохранить значения usage.
2. После ingest/chat обновить usage.

Ожидание:

- `used` увеличивается;
- `remaining` уменьшается;
- recent events заполняется.

## 7. Isolation checks

### I1. Раздельные workspace

Шаги:

1. Создать `user_a`.
2. Создать `user_b`.
3. Загрузить разные документы.
4. Открыть wiki и sessions под каждым пользователем.

Ожидание:

- `user_a` не видит документы, wiki и sessions `user_b`;
- `user_b` не видит документы, wiki и sessions `user_a`.

### I2. Cross-user negative checks

Шаги:

1. Под `user_b` попытаться открыть session id от `user_a`.
2. Под `user_b` попытаться открыть slug, которого нет в его workspace, но есть у
   `user_a`.
3. Под `user_b` вызвать usage и admin endpoints.

Ожидание:

- чужая session недоступна;
- чужая page недоступна;
- usage только собственный;
- admin endpoints недоступны.

## 8. Quota checks

### Q1. Usage event creation

Шаги:

1. Под `user_a` выполнить ingest.
2. Под `user_a` выполнить chat.
3. Проверить usage.

Ожидание:

- usage events создаются;
- значения usage меняются после обоих действий.

### Q2. Limit enforcement

Шаги:

1. Установить маленькие лимиты в тестовом окружении.
2. Выполнить несколько запросов до упора.

Ожидание:

- после исчерпания лимита backend блокирует дальнейшие запросы к LLM;
- пользователь получает понятную ошибку.

## 9. Personal mode sanity checks

Даже если alpha идет на `multi_user`, перед релизом нужно кратко проверить:

- `personal_local` стартует;
- `personal_server` стартует;
- Basic Auth в `personal_server` работает;
- personal режимы не требуют account auth.

## 10. Что фиксировать как баг

P0:

- видно чужие данные;
- не работает register/login/upload/ingest/chat;
- приложение падает;
- quota обходится;
- admin endpoint доступен обычному пользователю.

P1:

- usage считает неправильно;
- admin role не выдается;
- анонимный `multi_user` клиент не видит login screen;
- UI не дает пройти основной сценарий;
- conflict flow ломает workspace;
- ошибки непонятны пользователю.

P2:

- визуальные шероховатости;
- неудобный текст;
- мелкие UX недочеты;
- неполная локализация;
- cosmetic inconsistencies.

## 11. Формат обратной связи

Для каждого тестового прогона фиксировать:

```md
## Дата / Пользователь

### Сценарий
- ...

### Что сработало
- ...

### Что сломалось
- ...

### Класс
- P0 / P1 / P2

### Повторяемость
- always / sometimes / once
```

## 12. Критерий допуска ранних пользователей

Можно выдавать доступ alpha-пользователям, если:

- внутренний smoke пройден;
- admin/user/isolation checks пройдены;
- browser auth smoke для `multi_user` пройден;
- нет открытых P0;
- известные ограничения документированы;
- есть один staging URL и один понятный способ логина/регистрации.

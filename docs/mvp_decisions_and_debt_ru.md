# MVP Decisions and Debt

Дата: 2026-05-24

Этот документ фиксирует принятые инженерные решения для alpha-MVP и
технический долг, который сознательно оставлен за пределами текущего scope.

## 1. Принятые решения

### 1.1 Один кодовый контур для трех режимов

Оставляем одну кодовую базу для:

- `personal_local`;
- `personal_server`;
- `multi_user`.

Почему:

- это минимизирует расхождение поведения между режимами;
- упрощает поддержку и тестирование;
- позволяет развивать продукт без отдельного форка под каждый сценарий.

### 1.2 SQLite only для control plane в alpha

Для `multi_user` control plane хранится в SQLite.

Почему:

- этого достаточно для закрытого alpha и первых пользователей;
- это сохраняет простоту запуска;
- перенос на Postgres не нужен для проверки ценности продукта.

Компромисс:

- SQLite остается ограничением по concurrency и operational tooling;
- этот выбор фиксирован только для MVP, не для долгосрочной архитектуры.

### 1.3 Вики и документы живут в `wiki-data`

Основное состояние продукта хранится в файловом `wiki-data`.

Почему:

- это соответствует markdown-first подходу;
- данные легко переносить и инспектировать;
- нет лишней абстракции в виде отдельной БД для wiki.

### 1.4 Абсолютные persistent paths

Для контейнерного запуска используем только абсолютные пути:

- `CONTROL_DB_URL=sqlite:////wiki-data/control.db`;
- `WIKI_WORKSPACES_ROOT=/wiki-data/workspaces`;
- `WIKI_DATA_PATH=/wiki-data`.

Почему:

- относительные пути зависели от `WORKDIR` контейнера;
- из-за этого persistence мог тихо сломаться после rebuild;
- абсолютный путь делает поведение прозрачным.

### 1.5 Админ определяется через email из env

Админ не имеет дефолтного логина и пароля.

Почему:

- это убирает hardcoded credentials;
- bootstrap проще и безопаснее для alpha;
- админский аккаунт создается обычной регистрацией на разрешенный email.

### 1.6 `multi_user` использует встроенный login/register screen

В `multi_user` нет browser popup Basic Auth.

Почему:

- это позволяет нормальный user onboarding;
- UI остается единым;
- auth boundary реализуется на API уровне.

### 1.7 Запуск контейнера от непривилегированного пользователя

`wiki-engine` в контейнере запускается от `appuser` (uid 1000) через entrypoint.

Почему:

- bind-mounted `wiki-data` должен быть writable;
- entrypoint (`docker-entrypoint.sh`) при старте чинит права на `/wiki-data` и `/app`, затем через `gosu` дропает привилегии до `appuser`;
- это production-ready hardening без необходимости отдельного init-контейнера.

### 1.8 Кредитная система: дневной сброс только при использовании

Дневной лимит сбрасывается в 00:00 UTC только если пользователь фактически расходовал токены.

Правила:

- дневной и приветственный лимиты суммируются — пользователь видит общий остаток;
- сначала расходуется дневной лимит, затем приветственный;
- приветственный лимит не имеет ограничения по времени;
- если пользователь не тратил токены за день — лимит не сбрасывается, bucket остаётся нетронутым, reset_at сдвигается на следующий период.

Почему:

- пользователь, не использовавший систему, не теряет лимит;
- логика расхода (daily first → welcome) интуитивно понятна;
- приветственный бонус остаётся доступным пока не израсходован.

### 1.9 Unicode slugs с автоматической нормализацией

Slug (путь wiki-страницы) поддерживает Unicode-символы (кириллицу, иероглифы и т.д.).

Почему:

- LLM генерирует названия страниц на естественном языке;
- ограничение только на ASCII приводило к ошибкам валидации при клике на `[[wikilink]]`.

Нормализация через `slugify()`:

- приводит к нижнему регистру;
- заменяет пробелы, точки, запятые на `_`;
- заменяет неразрывный дефис (U+2011) на обычный `-`;
- любые другие недопустимые символы → `_`;
- сохраняет ведущий `_` (важно для `_general/`);
- применяется на этапе парсинга LLM-ответа (Step 1 и Step 2).

### 1.10 Песочница кода с AST-анализом и resource limits

Встроенный `CodeInterpreter` защищён тремя уровнями:

1. AST-анализ: перед выполнением код проверяется — запрещены опасные импорты (`os`, `subprocess`, `socket`, `requests`, `httpx`…), вызовы (`eval`, `exec`, `__import__`, `open(mode='w')`…) и другие уязвимые операции.
2. Resource limits (POSIX): CPU ограничен таймаутом (10 сек), память — 256 MB, размер создаваемых файлов — 1 MB, core dump отключён.
3. Таймаут subprocess: 10 секунд hard limit.

Почему:

- в multi-user режиме пользователи могут выполнять код через LLM-агентов;
- необходима защита от случайного или намеренного выполнения опасных операций;
- AST-анализ отсекает запрещённое до запуска, resource limits ограничивают ущерб после запуска.

### 1.11 Нормализация provenance-маркеров в post-processing

При ingest `normalize_wikilinks` автоматически исправляет:

- `^[path/file.ext]` → `^[raw/path/file.ext]` — добавляет префикс `raw/` если его нет;
- `^[raw/...[[wikilink]]...]` → `^[raw/...display...]` — удаляет вложенные wikilinks внутри provenance-маркеров.

Почему:

- LLM часто генерирует provenance-маркеры без префикса `raw/`;
- LLM иногда вставляет `[[wikilink]]` внутрь `^[...]`;
- линтер проверяет и те, и другие.

### 1.12 Линтер проверяет provenance без raw/

`WikiLinter._check_provenance` теперь детектирует:

- `^[path/file]` без `raw/` → warning с фикс-хинтом;
- битые `[[wikilink]]` → error (было и раньше).

### 1.13 Нормализация битых wikilink'ов с невалидными символами

При пост-обработке `normalize_wikilinks` конвертирует `[[slug]]` в plain text, если slug не проходит `is_safe_slug()` (содержит пробелы, кириллицу как часть имени файла вместо slug, нестандартные символы).

Почему:

- LLM часто генерирует `[[_general/Уточненное описание]]` вместо `^[raw/_general/Уточненное описание.docx]`;
- после strip extension эта ссылка не опознавалась как raw-источник;
- теперь любой `[[slug]]` с недопустимыми символами превращается в обычный текст, предотвращая битые ссылки при перестроении.

### 1.14 Структурный линтинг в UI

Добавлен `GET /api/audit/lint` — endpoint, возвращающий полный отчёт WikiLinter'а (17 проверок). В модалке «Аудит» появилась кнопка «Проверить».

## 2. Технический долг

### 2.1 Admin API для управления пользователями и лимитами

Долг:

- нет полноценного admin UI;
- нет удобного управления лимитами из интерфейса;
- нет панели для batch-операций по пользователям.

Почему не сейчас:

- MVP проверяет базовый сценарий и изоляцию;
- ручного bootstrap через env достаточно для первого теста.

### 2.2 Production-grade operations

Долг:

- backup/restore workflow;
- observability stack;
- rate limiting / anti-abuse;
- email verification;
- password reset;
- OAuth;
- Postgres migration.

Почему не сейчас:

- эти темы не нужны для первого закрытого тестирования;
- они не должны размывать MVP boundary.

## 3. Правило работы с долгом

Если новая задача:

- не закрывает security gap;
- не блокирует основной пользовательский сценарий;
- не нужна для закрытого теста;

то она уходит в post-MVP backlog, а не в текущий alpha scope.

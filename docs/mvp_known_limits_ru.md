# MVP Known Limits

Дата: 2026-05-22

Этот документ нужен для alpha-пользователей и внутренних тестировщиков. Его
задача - заранее обозначить ограничения MVP, чтобы не смешивать ожидаемое
поведение с реальными багами.

## 1. Статус версии

Текущая версия - закрытый alpha MVP.

Это означает:

- основной сценарий должен работать;
- безопасность tenant boundary должна соблюдаться;
- интерфейс и workflow еще могут быть шероховатыми;
- часть product-grade функций сознательно отложена.

## 2. Что является нормальным ограничением MVP

### Контур продукта

- система не является полноценной SaaS knowledge platform;
- роли ограничены `user` и `admin`;
- shared workspace между пользователями нет;
- billing и платная модель доступа отсутствуют;
- полноценной админ-панели управления пользователями нет.

### Хранение и инфраструктура

- wiki-контент хранится в файловой структуре;
- server control plane хранится в SQLite;
- у администратора нет default login/password: admin role выдается аккаунту,
  зарегистрированному на email из `MULTI_USER_ADMIN_EMAIL(S)`;
- система не рассчитана на высокую конкурентную нагрузку;
- нет отдельного background worker контура;
- нет production backup/restore workflow как готовой функции.

### LLM behavior

- качество ответа зависит от качества исходных документов;
- модель может отвечать неполно, если wiki-контекст слабый;
- ingest и query чувствительны к качеству выбранного LLM provider;
- очень слабые или нестабильные модели могут ломать качество анализа.

### Документы

- первый alpha лучше проверять на `.md`, `.txt`, `.py`;
- поддержка `.pdf`, `.docx`, `.pptx` есть, но качество зависит от извлечения
  текста;
- большие и плохо структурированные документы могут давать менее стабильный
  ingest, чем компактные markdown-источники;
- при исчерпании дневного лимита токенов ингест сохраняет частичные результаты
  (обработанные чанки) и останавливается;
- ингест выполняется в фоновом потоке и не блокирует другие запросы к приложению;
- отменить активный ингест можно через `POST /api/ingest/cancel`.

### UI

- UI ориентирован на рабочий сценарий, а не на polished production UX;
- в `multi_user` вход идет через встроенный login/register screen, а не через
  browser popup;
- отдельные тексты и сообщения могут быть еще не идеальны;
- часть админского опыта пока идет через API, а не через полноценный интерфейс.

## 3. Что не считается багом alpha

- отсутствие billing;
- отсутствие OAuth;
- отсутствие email verification и password reset;
- отсутствие команд, организаций и shared workspace;
- отсутствие ролей сложнее `user/admin`;
- отсутствие расширенной аналитики;
- отсутствие production-grade observability stack;
- отсутствие Postgres;
- необходимость использовать тестовые документы в первом прогоне;
- заметная зависимость качества ответа от качества LLM.

## 4. Что уже считается багом

### P0

- пользователь видит чужие данные;
- обычный пользователь получает доступ к `/api/admin/*`;
- quota не ограничивает LLM usage;
- register/login/logout ломаются;
- ingest или chat ломают основной сценарий;
- приложение не стартует по runbook.

### P1

- usage отображается некорректно;
- admin email не получает admin role;
- анонимный `multi_user` клиент попадает в основной UI без login screen;
- workspace изоляция нестабильна;
- UI не дает пройти основной сценарий без ручного обхода;
- ошибки непонятны настолько, что блокируют тестирование.

### P2

- неидеальная верстка;
- шероховатые тексты;
- неудобства, не блокирующие сценарий;
- minor visual inconsistencies.

## 5. Ограничения первого внешнего тестирования

Ранним пользователям не рекомендуется:

- загружать чувствительные документы до отдельного security review;
- проверять систему на больших пакетах PDF в первом цикле;
- использовать десятки пользователей одновременно;
- делать вывод о long-term scalability по alpha-версии.

Рекомендуется:

- загрузить 3-5 документов одной темы;
- задать 5-10 предметных вопросов;
- отметить ошибки ответа и UX блокеры;
- проверить, понятен ли workflow без помощи разработчика.

## 6. Ограничения по запуску

- `multi_user` требует заполненного `.env`;
- для `docker compose` внешний порт задается через `APP_PORT`, по умолчанию
  `8000`;
- staging запуск требует persistent storage для `wiki-data`;
- при неверном `LLM_API_KEY` приложение может стартовать, но LLM сценарии будут
  недоступны;
- после изменения env нужен рестарт приложения.

## 7. Ограничения по безопасности

Текущий security baseline:

- tenant boundary обязателен;
- admin API отделен;
- auth обязателен для `multi_user`;
- workspace изолированы.

Но пока нет:

- полноценного внешнего security audit;
- hardened production secrets management;
- расширенного audit trail;
- enterprise access model;
- rate limiting и anti-abuse контура production-класса.

## 8. Решение по ожиданиям

Правильное ожидание от alpha MVP:

- система должна быть достаточно стабильной, чтобы пройти реальный сценарий;
- система еще не обязана быть полной, идеально удобной или рассчитанной на
  production-нагрузку;
- обратная связь должна идти в первую очередь по качеству workflow, wiki,
  ingest, query и isolation, а не по отсутствующим post-MVP функциям.

# Wiki Engine

Персональная wiki на основе LLM из markdown-документов. Загружает исходные документы (`.md`, `.txt`, `.py`, `.pdf`, `.docx`, `.pptx`), строит структурированную базу знаний с перекрёстными ссылками, находит конфликты между источниками, накапливает переиспользуемые правила и отвечает на вопросы через LLM. **Без баз данных. Без векторных эмбеддингов.** Всё хранится в виде обычных текстовых файлов.

> **Вдохновлено** [gist Андрея Карпати (Andrej Karpathy)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) — идея LLM-поддерживаемой структурированной wiki из сырых документов.

## Что делает система

Вы кладёте документы в папку — система анализирует их, создаёт wiki-страницы, связывает их перекрёстными ссылками, находит противоречия и позволяет задавать вопросы по всей базе знаний.

## Как пользоваться — пошагово

### 1. Запустите сервер
```powershell
python -m uvicorn app.api.main:app --reload --port 8000
```
Откройте `http://localhost:8000`.

### 2. Загрузите документы
Нажмите **📂 Загрузить** в верхней панели. Выберите проект и файлы. Система автоматически:
— Извлечёт текст (включая из PDF, DOCX, PPTX)
— Проанализирует содержимое (Step 1)
— Создаст wiki-страницы с перекрёстными ссылками (Step 2)
— Проверит структуру линтером

### 3. Задавайте вопросы
В центральном окне чата пишите вопросы по-русски (или на любом языке). Система найдёт релевантные страницы и ответит со ссылками `[[slug]]`.

Фильтр проектов вынесен в отдельный селектор рядом с полем ввода: можно выбрать несколько проектов и быстро сбросить фильтр.

### 4. Разрешайте конфликты
Если в документах обнаружены противоречия, они появятся во вкладке **Конфликты** (правая панель). Процесс разрешения:
1. Понять суть противоречия
2. Выбрать интерпретацию (варианты предложены)
3. Просмотреть извлечённый навык (skill)
4. Подтвердить — система применит изменения и запишет правило в `skills.md`

### 5. Перестройте wiki при необходимости
Кнопка **🔄 Перестроить** заново обрабатывает все исходные файлы из `raw/`. Полезно после:
— Изменения исходных документов
— Обновления `skills.md` или `AGENTS.md`
— Смены LLM-модели

### 6. Проверьте дубликаты
Кнопка **🔍 Аудит** запускает проверку на дублирующиеся и пересекающиеся страницы. По умолчанию показывает только точные дубликаты (одинаковые названия в одном проекте). Кнопка **+ Пересечения** раскрывает страницы с общими ссылками и тегами.

## Интерфейс

Интерфейс разделён на три зоны:

### Левая панель — сессии и управление
- **История чатов**: переключение между сессиями, удаление (✕)
- **Кнопки хедера**: 📂 Загрузить, 🔍 Аудит, меню **Управление** с действиями перестроения и очистки

### Центр — чат
- Ввод вопросов по Enter
- **Фильтр проектов**: выпадающий список с мультивыбором
- **Цитаты**: `[[slug]]` — клик открывает страницу в правой панели
- **Provenance**: маркеры `^[raw/file.md]` ведут к исходным документам
- В multi-user режиме provenance-ссылки и прямое скачивание raw-файлов используют `access_token` автоматически; если открыть raw-ссылку вручную без авторизации, сервер вернёт 401
- Контекст сессии сохраняется между сообщениями

### Правая панель — wiki-навигатор
- **Wiki**: дерево проектов, просмотр страниц, полнотекстовый поиск
- **Конфликты**: открытые, решённые, межпроектные — с пошаговым разрешением
- **Черновики**: ожидающие применения изменения

## Ключевые возможности

### 1. Мультиформатная загрузка
Поддерживается 6 форматов: `.md`, `.txt`, `.py`, `.pdf`, `.docx`, `.pptx`. Бинарные форматы конвертируются через markitdown.

### 2. Двухшаговый ingest (Plan → Execute)
- **Шаг 1 (Анализ)**: LLM читает источник, выделяет сущности/концепты, находит конфликты, планирует страницы
- **Шаг 2 (Генерация)**: LLM создаёт wiki-страницы с frontmatter, wikilinks, источниками и provenance-маркерами

Каждая страница содержит структурированный frontmatter: `title`, `project`, `type`, `tags`, `confidence`, `sources`, `last_confirmed`.

### 3. Автоматическая пост-обработка
После генерации система исправляет артефакты LLM:
- Удаляет вложенные wikilinks (`[[a/[[b]]` → `[[b]]`)
- Заменяет обратную косую черту в путях (`\` → `/`)
- Убирает ссылки на raw-файлы из wikilinks
- Проверяет и чинит `project` в frontmatter

### 4. Конфликты и навыки
При обнаружении противоречий:
- Конфликт записывается в `conflicts.md` с полным контекстом
- **Межпроектные различия** идентифицируются отдельно — не считаются ошибкой
- При разрешении извлекается правило (skill) и добавляется в `skills.md`

### 5. Накопление навыков (skills)
Разрешённые конфликты порождают переиспользуемые правила:
```
Source Trust Rules:
- Для eywa-demo: deploy_guide.md — авторитетный источник по инфраструктуре
Conflict Resolution Patterns:
- Управление зависимостями и безопасность — разные зоны ответственности
```

### 6. Структурный линтер
Автоматически после каждого ingest проверяет: битые wikilinks, потерянные страницы, дубликаты заголовков, устаревшие страницы, превышение лимитов, отсутствующий frontmatter, неверные provenance-маркеры.

### 7. Аудит дубликатов, пересечений и структуры
Кнопка **🔍 Аудит** запускает модалку с двумя разделами:

**Пересечения/дубликаты** (`GET /api/audit/duplicates`):
- **Дубликаты** (score ≥ 0.9): одинаковый заголовок в одном проекте
- **Пересечения** (score ≥ 0.6): общие wikilinks в пределах проекта
- **Теги** (score ≥ 0.5): 3+ общих тегов в одном проекте

**Структурный линтинг** (`GET /api/audit/lint`): 17 проверок без LLM — битые wikilinks, отсутствующий frontmatter, превышение лимитов, осиротевшие страницы, невалидные provenance-маркеры, дубликаты заголовков и др.

### 8. Raw-файлы и provenance
- Страницы и чата содержат provenance-маркеры вида `^[raw/path/to/file.md]`
- В multi-user режиме UI подставляет `access_token` к raw-ссылкам автоматически, чтобы скачивание работало без отдельного ручного логина
- Эндпоинт raw-скачивания: `GET /api/wiki/raw/{slug}`.

### 9. ReAct-агент запросов
Вопросы классифицируются (фактические/сравнительные/исследовательские). Агент находит релевантные страницы, отвечает на русском, цитирует источники через `[[slug]]` и честно сообщает о неуверенности.

### 10. Кросс-проектная поддержка
- Документы распределены по проектам (`eywa-demo`, `aurora-demo`, `_general`)
- Межпроектные страницы (в `_general/`) сравнивают подходы без ложных конфликтов
- Фильтр проектов в чате и поиске

### 11. Wiki-ссылки (wikilinks)
При ingest страницы автоматически связываются через `auto_link` (до 10 кандидатов). Формат: `[[slug]]` или `[[slug|текст]]`.

## Быстрый старт

### Windows (нативно)

1. **Клонируйте и установите:**
   ```powershell
   git clone https://github.com/aikula/ai_personal_wiki.git
   cd ai_personal_wiki
   pip install -e .
   ```

2. **Создайте файл `.env`**:
   ```powershell
   Copy-Item .env.example .env
   ```

3. **Запустите сервер:**
   ```powershell
   python -m uvicorn app.api.main:app --reload --port 8000
   ```

4. **Откройте браузер:** http://localhost:8000

5. **Загрузите документы** (📂) → **Задайте вопросы** → **Разрешите конфликты** → **Перестройте** при необходимости

### Linux / macOS

```bash
git clone https://github.com/aikula/ai_personal_wiki.git
cd ai_personal_wiki
pip install -e .
cp .env.example .env
uvicorn app.api.main:app --reload --port 8000
```

### Docker

```bash
cp .env.example .env
# Отредактируйте .env — добавьте свои учётные данные
docker compose up --build
```

Откройте `http://localhost:8000`.

> **Windows + Docker:** Docker Desktop с томами в OneDrive может работать медленно. Используйте нативный запуск (Вариант 1), если `wiki-data` на OneDrive.

## Устранение неполадок

### После перезапуска сервера аудит показывает старые данные
**Причина:** браузер кеширует index.html.
**Решение:** жёсткая перезагрузка страницы (Ctrl+F5).

### LLM возвращает `unsupported_country_region_territory` (403)
**Причина:** OpenAI API недоступен в вашем регионе.
**Решение:** используйте локальный LLM через Ollama или другой endpoint.

### Ошибка `Could not parse JSON from LLM response`
**Причина:** LLM не уложился в лимит `max_tokens`.
**Решение:** используйте модель с большим контекстным окном или уменьшите `entity_page_chars` / `concept_page_chars` в `config/settings.yaml`.

### Страницы создаются с неправильным проектом
**Решение:** запустите `🔄 Перестроить` — после перестройки `project` в frontmatter будет исправлен автоматически.

## Мульти-пользовательский режим (multi-user)

Wiki Engine поддерживает три режима (`APP_MODE`):

| Режим | Описание |
|---|---|
| `personal_local` | Локальная установка, без аутентификации, без квот |
| `personal_server` | Серверная установка с Basic Auth (`WIKI_AUTH_ENABLED=true`) |
| `multi_user` | Мульти-пользователь: регистрация, Bearer-токены, квоты токенов, изолированные workspace'ы |

В multi-user режиме:
- **Регистрация**: `POST /api/auth/register` — создаёт пользователя, workspace и кредитные корзины
- **Квоты**: дневной лимит (`DEFAULT_DAILY_TOKENS`) + приветственный (`DEFAULT_WELCOME_TOKENS`), отслеживание через `MeteredLLMClient`
- **Изоляция**: каждый пользователь получает отдельный workspace в `/wiki-data/workspaces/{user_id}/`
- **SQLite control plane**: учётные записи, сессии, квоты — в `/wiki-data/control.db` (миграции в `app/core/migrations/`)
- **Администраторы**: email'ы из `MULTI_USER_ADMIN_EMAILS` получают права администратора при создании

## Безопасность песочницы кода

Встроенный интерпретатор (`CodeInterpreter`) запускает код в изолированном процессе с тремя уровнями защиты:

1. **AST-анализ**: перед запуском код проверяется — запрещены опасные импорты (`os`, `subprocess`, `socket`, `requests`…), вызовы (`eval`, `exec`, `__import__`, `open(mode='w')`…) и другие уязвимые операции
2. **Лимиты ресурсов (POSIX)**: CPU ограничен таймаутом, память — 256 MB, размер создаваемых файлов — 1 MB
3. **Таймаут**: 10 секунд hard limit на выполнение

Docker-контейнер запускается от непривилегированного пользователя (`appuser`, uid 1000).

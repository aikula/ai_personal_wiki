# Enterprise Knowledge Compiler

> [English version](README.en.md)

**Enterprise Knowledge Compiler** — open-source система на основе LLM, которая превращает сырые документы в структурированную, проверяемую и пригодную для AI базу знаний.

Система загружает исходные документы (`.md`, `.txt`, `.py`, `.pdf`, `.docx`, `.pptx`), строит связанные wiki-страницы, сохраняет provenance до исходников, находит противоречия, накапливает переиспользуемые правила разрешения и отвечает на вопросы через LLM с цитированием. В публичной версии слой знаний хранится как обычные текстовые файлы, без обязательной базы данных и без обязательных векторных эмбеддингов.

> **Вдохновлено** [gist Андрея Карпати (Andrej Karpathy)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) — идеей LLM-поддерживаемой структурированной wiki из сырых документов.

## Позиционирование

Это не очередной "чат с документами". Обычный RAG чаще всего ищет фрагменты внутри уже существующего документного хаоса. Enterprise Knowledge Compiler сначала пытается скомпилировать этот хаос в управляемый слой знаний:

- исходные документы превращаются в структурированные страницы;
- страницы получают стабильные slug, frontmatter, ссылки и provenance;
- противоречия становятся явными конфликтами;
- экспертные решения превращаются в переиспользуемые правила в `skills.md`;
- структурный аудит находит дубликаты, пересечения, битые ссылки, устаревшие факты и ошибки provenance;
- полученная база знаний может использоваться людьми, ассистентами и AI-агентами.

## Редакции

| Редакция | Статус | Для чего | Хранилище / control plane |
|---|---|---|---|
| Open-source local edition | Публичный репозиторий | Личные исследования, локальная база знаний, демо | Plain files в `wiki-data/` |
| Open-source server edition | Публичный репозиторий | Частный VPS или небольшая командная демонстрация | Plain files, optional Basic Auth |
| Open-source multi-user edition | Публичный репозиторий | Небольшой hosted demo, изолированные workspace, учет токенов | Plain files + SQLite control plane |
| Corporate implementation | Платный пилот / адаптация / rollout | Документоемкие и регулируемые организации | Plain knowledge layer + PostgreSQL control plane, SSO/RBAC, audit, connectors, review workflows |

Корпоративная версия не описывается как полностью готовый коробочный SaaS. Это более жесткая реализация той же архитектурной идеи, которая адаптируется через платный пилот под документы, инфраструктуру, безопасность и процессы заказчика.

## Что делает система

Вы загружаете документы. Система анализирует их, создает wiki-страницы, связывает их перекрестными ссылками, находит противоречия и позволяет задавать вопросы по всей сгенерированной базе знаний.

Возможности публичной версии:

- мультиформатная загрузка;
- двухшаговый LLM-процесс: сначала анализ, затем генерация страниц;
- wiki-страницы со структурированным frontmatter;
- provenance-маркеры со ссылками на raw-файлы;
- поиск и ручное разрешение конфликтов;
- правила, извлекаемые из решенных конфликтов;
- структурный линтинг и аудит;
- разделение знаний по проектам;
- local/server/multi-user режимы.

## Корпоративный roadmap

Для корпоративных заказчиков проект может быть расширен до enterprise implementation:

- PostgreSQL control plane для пользователей, workspace, audit events, permissions, usage и job metadata;
- SSO/OIDC/SAML или интеграция с корпоративной identity-системой;
- RBAC и командные права доступа;
- pipeline для больших объемов документов;
- Source Cards, Claims Layer, source drift tracking и более строгий provenance;
- экспертные review workflows для смысловых изменений;
- connectors к SharePoint, Google Drive, Confluence, Jira, file shares, object storage, DMS/ECM;
- deployment на инфраструктуре заказчика, private cloud или контролируемом региональном cloud;
- observability, job queues, backups и operational runbooks;
- Arabic/English демо-наборы и доменные пилоты.

См. [`docs/corporate_edition_roadmap.md`](docs/corporate_edition_roadmap.md).

## Как пользоваться — пошагово

### 1. Запустите сервер
```powershell
python -m uvicorn app.api.main:app --reload --port 8000
```
Откройте `http://localhost:8000`.

### 2. Загрузите документы
Нажмите **📂 Загрузить** в верхней панели. Выберите проект и файлы. Система автоматически:
— Извлечет текст (включая из PDF, DOCX, PPTX)
— Проанализирует содержимое (Step 1)
— Создаст wiki-страницы с перекрестными ссылками (Step 2)
— Проверит структуру линтером

### 3. Задавайте вопросы
В центральном окне чата пишите вопросы. Система найдет релевантные страницы и ответит со ссылками `[[slug]]`.

Фильтр проектов вынесен в отдельный селектор рядом с полем ввода: можно выбрать несколько проектов и быстро сбросить фильтр.

### 4. Разрешайте конфликты
Если в документах обнаружены противоречия, они появятся во вкладке **Конфликты** (правая панель). Процесс разрешения:
1. Понять суть противоречия
2. Выбрать интерпретацию (варианты предложены)
3. Просмотреть извлеченный навык (skill)
4. Подтвердить — система применит изменения и запишет правило в `skills.md`

Для обновления wiki-страницы по разрешённому конфликту:
1. **Prepare update**: генерирует кандидат через LLM, показывает diff
2. **Apply update**: применяет сохранённый кандидат (НЕ вызывает LLM повторно)
3. **Reject update**: удаляет черновик без изменений страницы

### 5. Перестройте wiki при необходимости
Кнопка **🔄 Перестроить** заново обрабатывает все исходные файлы из `raw/`. Полезно после:
— Изменения исходных документов
— Обновления `skills.md` или `AGENTS.md`
— Смены LLM-модели

### 6. Проверьте дубликаты
Кнопка **🔍 Аудит** запускает проверку на дублирующиеся и пересекающиеся страницы. По умолчанию показывает только точные дубликаты (одинаковые названия в одном проекте). Кнопка **+ Пересечения** раскрывает страницы с общими ссылками и тегами.

## Интерфейс

Интерфейс разделен на три зоны:

### Левая панель — сессии и управление
- **История чатов**: переключение между сессиями, удаление (✕)
- **Кнопки хедера**: 📂 Загрузить, 🔍 Аудит, меню **Управление** с действиями перестроения и очистки

### Центр — чат
- Ввод вопросов по Enter
- **Фильтр проектов**: выпадающий список с мультивыбором
- **Цитаты**: `[[slug]]` — клик открывает страницу в правой панели
- **Provenance**: маркеры `^[raw/file.md]` ведут к исходным документам
- В multi-user режиме provenance-ссылки и прямое скачивание raw-файлов используют `access_token` автоматически
- Контекст сессии сохраняется между сообщениями

### Правая панель — wiki-навигатор
- **Wiki**: дерево проектов, просмотр страниц, полнотекстовый поиск
- **Конфликты**: открытые, решенные, межпроектные — с пошаговым разрешением
- **Черновики**: ожидающие применения изменения

## Ключевые возможности

### 1. Мультиформатная загрузка
Поддерживается 6 форматов: `.md`, `.txt`, `.py`, `.pdf`, `.docx`, `.pptx`. Бинарные форматы конвертируются через markitdown.

### 2. Двухшаговый ingest (Plan → Execute)
- **Шаг 1 — Анализ**: LLM читает источник, выделяет сущности/концепты, находит конфликты, планирует страницы
- **Шаг 2 — Генерация**: LLM создает wiki-страницы с frontmatter, wikilinks, источниками и provenance-маркерами

Каждая страница содержит структурированный frontmatter: `title`, `project`, `type`, `tags`, `confidence`, `sources`, `last_confirmed`.

### 3. Автоматическая пост-обработка
После генерации система исправляет артефакты LLM:
- Удаляет вложенные wikilinks (`[[a/[[b]]` → `[[b]]`)
- Заменяет обратную косую черту в путях (`\` → `/`)
- Убирает ссылки на raw-файлы из wikilinks
- Проверяет и чинит `project` в frontmatter

### 4. Конфликты и навыки
При обнаружении противоречий:
- Конфликт записывается в `conflicts.md` с полным контекстом
- **Межпроектные различия** идентифицируются отдельно — не считаются ошибкой
- При разрешении извлекается правило (skill) и добавляется в `skills.md`

### 5. Накопление навыков
Разрешенные конфликты порождают переиспользуемые правила:
```
Source Trust Rules:
- Для eywa-demo: deploy_guide.md — авторитетный источник по инфраструктуре
Conflict Resolution Patterns:
- Управление зависимостями и безопасность — разные зоны ответственности
```

### 6. Структурный линтер
Автоматически после каждого ingest проверяет: битые wikilinks, потерянные страницы, дубликаты заголовков, устаревшие страницы, превышение лимитов, отсутствующий frontmatter, неверные provenance-маркеры.

### 7. Аудит дубликатов, пересечений и структуры

Кнопка **🔍 Аудит** запускает модалку с двумя разделами:

**Пересечения/дубликаты** (`GET /api/audit/duplicates`):
- **Дубликаты** (score ≥ 0.9): одинаковый заголовок в одном проекте
- **Пересечения** (score ≥ 0.6): общие wikilinks в пределах проекта
- **Теги** (score ≥ 0.5): 3+ общих тегов в одном проекте

**Структурный линтинг** (`GET /api/audit/lint`): 17 проверок без LLM — битые wikilinks, отсутствующий frontmatter, превышение лимитов, осиротевшие страницы, невалидные provenance-маркеры, дубликаты заголовков и др.

### 8. Raw-файлы и provenance
- Страницы и чат содержат provenance-маркеры вида `^[raw/path/to/file.md]`
- В multi-user режиме UI подставляет `access_token` к raw-ссылкам автоматически
- Эндпоинт raw-скачивания: `GET /api/wiki/raw/{slug}`

### 9. ReAct-агент запросов
Вопросы классифицируются (фактические/сравнительные/исследовательские/meta). Агент находит релевантные страницы, цитирует источники через `[[slug]]` и честно сообщает о неуверенности.

### 10. Кросс-проектная поддержка
- Документы распределены по проектам (`eywa-demo`, `aurora-demo`, `_general`)
- Межпроектные страницы (в `_general/`) сравнивают подходы без ложных конфликтов
- Фильтр проектов в чате и поиске

### 11. Wiki-ссылки
При ingest страницы автоматически связываются через `auto_link` (до 10 кандидатов). Формат: `[[slug]]` или `[[slug|текст]]`.

## Мульти-пользовательский режим

Wiki Engine поддерживает три режима (`APP_MODE`):

| Режим | Описание |
|---|---|
| `personal_local` | Локальная установка, без аутентификации, без квот |
| `personal_server` | Серверная установка с Basic Auth (`WIKI_AUTH_ENABLED=true`) |
| `multi_user` | Мульти-пользователь: регистрация, Bearer-токены, квоты токенов, изолированные workspace |

В multi-user режиме:
- **Регистрация**: `POST /api/auth/register` — создает пользователя, workspace и кредитные корзины
- **Квоты**: дневной лимит (`DEFAULT_DAILY_TOKENS`) + приветственный (`DEFAULT_WELCOME_TOKENS`), отслеживание через `MeteredLLMClient`
- **Изоляция**: каждый пользователь получает отдельный workspace
- **SQLite control plane**: учетные записи, сессии, квоты — в `control.db`

## API Endpoints

### Ingest
| Метод | Путь | Описание |
|---|---|---|
| `POST` | `/api/ingest` | Загрузка одного файла (form-data: `project` + `file`) |
| `POST` | `/api/ingest/batch` | Массовая загрузка (form-data: `project` + `files[]` или `file[]`). Пустой батч → 400 |
| `POST` | `/api/ingest/rebuild` | Полная перестройка wiki из raw (SSE) |
| `POST` | `/api/ingest/clear` | Сброс wiki в исходное состояние |
| `GET` | `/api/ingest/raw` | Список raw-файлов |
| `GET/POST` | `/api/ingest/drafts/*` | Работа с черновиками |

### Chat
| Метод | Путь | Описание |
|---|---|---|
| `POST` | `/api/chat` | Отправить вопрос (SSE-поток) |
| `GET` | `/api/chat/sessions` | Список сессий |
| `GET/DELETE` | `/api/chat/sessions/{id}` | Получить / удалить сессию |
| `POST` | `/api/chat/sessions/{id}/crystallize` | Сохранить сессию как wiki-страницу |

### Wiki
| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/wiki/tree` | Дерево wiki для навигации |
| `GET` | `/api/wiki/page/{slug}` | Отрендерить страницу |
| `GET` | `/api/wiki/search` | Полнотекстовый поиск |
| `GET` | `/api/wiki/projects` | Список проектов |
| `GET` | `/api/wiki/raw/{path}` | Исходный файл из `raw/` для просмотра/скачивания (`access_token` поддерживается в multi-user) |

### Conflicts
| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/conflicts` | Список конфликтов |
| `GET` | `/api/conflicts/{id}` | Детали конфликта |
| `POST` | `/api/conflicts/{id}/resolve` | Разрешить конфликт |
| `POST` | `/api/conflicts/{id}/prepare-update` | Подготовить черновик |
| `POST` | `/api/conflicts/{id}/apply-update` | Применить разрешение |

### Audit
| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/audit/duplicates` | Поиск дубликатов и пересечений |
| `GET` | `/api/audit/lint` | Полный структурный линтинг |
| `GET` | `/api/audit/synthesis` | Список кандидатов на слияние |
| `POST` | `/api/audit/synthesis` | Запустить синтез |
| `POST` | `/api/audit/synthesis/{cid}/resolve` | Применить слияние |
